如果要将你的上游修改为**异步模式**以适配 NewAPI 的视频接口（`/v1/video/generations`），你需要实现**两个阶段**的接口逻辑。

NewAPI 遵循的是类 OpenAI/Luma 的异步标准。以下是上游服务器需要遵循的规范：

---

### 第一阶段：创建任务
**接口：** `POST /v1/video/generations`

当 NewAPI 请求你的上游时，你的上游必须**立即（在 1 秒内）**返回一个包含任务 ID 的响应，而不要等待视频生成完成。

**上游应返回的 JSON 格式：**
```json
{
    "id": "seedream-task-78901234", // 必填：你生成的唯一任务ID
    "model": "Seedream 4.0",        // 建议：返回请求的模型名
    "status": "pending",            // 必填：状态可以是 pending 或 processing
    "created_at": 1710650000        // 可选：秒级时间戳
}
```
*注意：一旦返回这个 JSON，NewAPI 就会认为任务提交成功，并开始根据这个 `id` 进行轮询。*

---

### 第二阶段：轮询查询任务状态
**接口：** `GET /v1/video/generations/{id}`

NewAPI 会定期（例如每隔 1-5 秒）调用这个接口，直到任务状态变为 `succeeded` 或 `failed`。

#### 情况 A：任务处理中
**上游应返回：**
```json
{
    "id": "seedream-task-78901234",
    "status": "processing",         // 必须是 processing
    "progress": 45                  // 可选：生成进度百分比
}
```

#### 情况 B：任务生成成功（完成）
**上游应返回：**
```json
{
    "id": "seedream-task-78901234",
    "status": "succeeded",          // 必须是 succeeded
    "video": {
        "url": "http://192.168.110.30/outputs/video_123.mp4" // 视频的最终可访问地址
    },
    "model": "Seedream 4.0"
}
```

#### 情况 C：任务失败
**上游应返回：**
```json
{
    "id": "seedream-task-78901234",
    "status": "failed",             // 必须是 failed
    "error": {
        "code": "gpu_out_of_memory",
        "message": "显存不足，生成失败"
    }
}
```

---

### 上游逻辑修改建议：

1.  **数据库/缓存**：你需要一个简单的数据库（如 Redis 或 SQLite）来存储 `id` 和对应的状态。
2.  **消息队列**：
    *   当 `POST` 请求进来时，将参数（prompt, ratio等）丢进一个异步队列。
    *   立即返回 `id`。
    *   后台 Worker 慢慢处理队列，并在完成后更新数据库里的 `status` 和 `url`。
3.  **URL 结构**：
    *   确保你的上游服务器能够解析带有路径变量的 GET 请求：`/v1/video/generations/任务ID`。

---

### NewAPI 侧的配置：

一旦你修改了上游，NewAPI 这边的配置就很简单了：
1.  **渠道类型**：选择 **“自定义渠道”**。
2.  **代理地址**：填 `http://192.168.110.30:5566`。
3.  **参数覆盖**：如果上游还需要 `ratio` 等参数，保留之前的 `copy` 模板：
    ```json
    {
      "operations": [
        { "mode": "copy", "from": "extra_body.ratio", "path": "ratio" },
        { "mode": "copy", "from": "extra_body.style", "path": "style" },
        { "mode": "delete", "path": "extra_body" }
      ]
    }
    ```

### 为什么必须这样改？
NewAPI 的视频模块代码是这样写的：
1.  调用下游创建接口。
2.  解析 Response Body，寻找 `id`。**如果没有 `id` 字段，直接判定为 400 Bad Request。**
3.  拿到 `id` 后，启动一个后台协程，定时访问查询接口。

**总结：** 如果你不想大改上游代码实现异步，最快的办法还是我之前说的——**假装是图片接口 (`/v1/images/generations`)**。如果你追求标准和稳定，那就按上面的异步规范修改。
