计划: 让图片/视频请求通过 OpenAI 兼容渠道透传
问题描述
你的上游 API 有三个端点，但只有对话 (/v1/chat/completions) 完全兼容 OpenAI 格式。图片 (/v1/images/generations) 和视频 (/v1/video/generations) 的请求和响应格式都是非标准的：

能力	上游请求格式	上游响应格式	new-api 期望的请求格式	new-api 期望的响应格式
对话	✅ OpenAI 标准	✅ OpenAI 标准	OpenAI 标准	OpenAI 标准
图片	❌ 含 ratio/style，无 size/quality/n	❌ 返回 chat.completion 格式	OpenAI 
ImageRequest
（需 size, n）	OpenAI 
ImageResponse
（data[].url）
视频	❌ 含 ratio/image，无 model	❌ 返回 chat.completion 格式	Task 系统（需绑定平台适配器）	Task 格式
可选方案分析
方案 A：在上游 API（你自己的服务）做适配 ⭐ 推荐
最小改动、最安全、不需要改 new-api 源码

原理：让你的上游 API 完全兼容 OpenAI 格式，new-api 当做普通的 OpenAI 类渠道处理。

图片方面：

你的上游 /v1/images/generations 接收标准 OpenAI 
ImageRequest
（含 prompt, model, n, size），内部映射 size -> ratio
返回标准 
ImageResponse
 格式：{"data": [{"url": "..."}], "created": ...}
你自己的 ratio/style 参数可以通过 OpenAI SDK 的 extra_body 穿透（dto.ImageRequest.Extra 字段已支持接收任意额外 JSON 字段）
视频方面：

视频不走 /v1/video/generations（这是 new-api 的异步 Task 系统，需要绑定特定平台适配器）
改为走 /v1/chat/completions 端点，model 使用如 doubao-video
你的上游收到 messages 后识别为视频请求，生成视频后返回标准 chat.completion 响应（content 里包含视频链接）
new-api 按普通 Chat 处理，用 Token 计费
方案 B：在 new-api 中新写自定义适配器 (ClawCloud)
上次对话已规划但未实施，改动点较多

需要修改 new-api 后端约 6~8 个文件来新增 ClawCloud 渠道类型，让 new-api 能原样透传非标格式请求。

方案 C：混合方案
图片在上游做兼容，视频走 Chat 透传

图片：上游返回标准 
ImageResponse
，或者客户端用 extra_body 传 ratio/style（已支持）
视频：统一走 /v1/chat/completions + 特定 model 名
推荐方案: 方案 A（上游做适配）+ 小量 new-api 调整
